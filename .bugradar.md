- The `integration_spotlight` test is intentionally marked `#[ignore]` and is not part of default CI or release workflows. This is a deliberate tradeoff: real Spotlight metadata behavior depends on host-specific services, indexing state, filesystem location, and local permissions, which makes the test non-deterministic in GitHub-hosted runners and unreliable as a release gate. We still run this test locally as a pre-release check (tag push hook), while keeping CI/release pipelines focused on deterministic checks.

- The `--print0` output shape (single-file: `origin\0`, multi-file: `path\0origin\0`) is also intentional and follows common Unix CLI conventions where context prefixes are omitted when only one input is provided (similar to grep-style behavior). This is documented in the README and treated as a compatibility choice for human/script ergonomics. For workflows that require a schema that is fully shape-stable regardless of input cardinality, `--jsonl` should be used instead.

- The `--jsonl` mode intentionally serializes file paths as UTF-8 text using lossy conversion semantics (`to_string_lossy` behavior). This means non-UTF-8 path bytes may be replaced rather than preserved byte-for-byte. This is an explicit format tradeoff for JSON interoperability and is documented behavior, not an accidental regression.

- When `kMDItemWhereFroms` is present but contains malformed or non-string values, the current behavior is to silently skip unusable entries and continue as if no usable origin was found. This is intentional in the current 0.1.x scope: the CLI prioritizes simple, stable default output over additional diagnostic modes or warning noise. Improving this with optional debug/verbose reporting is a possible future enhancement, but not required for current behavior correctness.

- Direct black-box coverage of `src/main.rs` mixed-result control flow (partial success, combined stdout/stderr, and non-BrokenPipe write failures) is intentionally limited in normal CI. This is an accepted risk for now: parser/printer/origins logic is covered by unit tests, and real end-to-end behavior is exercised as a pre-release gate via the local tag-push hook running `integration_spotlight`. Additional dedicated `main` I/O-path tests can be added later if regressions start appearing in this area.
